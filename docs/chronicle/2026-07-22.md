## 13:41 — Four knowledge-commons bugs, the patch mechanism, and a feature built at the wrong layer
**Branch:** worktree-session-tier
**Model:** opus-4.8
**Span:** 2026-07-21 → 2026-07-22

### Chronicle

#### Decisions

- **Patches propagate as semantic instructions anchored on section headings, never as diffs or regeneration.** Generated skills are project-owned and told to diverge; measurement showed headings survive generation byte-identical across all three graphs while bodies diverge up to 78%. That asymmetry is the whole design. Shipped as `/graph-patch` in v0.3.0.
- **Each delta's `satisfied-test` does double duty** — answering "is this already present?" before an edit and "did it land?" after. One artifact, both directions, on the reasoning that a mechanism built to end silent failure must not have that failure mode itself.
- **`Write` was revoked from `/graph-patch`'s allowed-tools** rather than only instructing against whole-file replacement. An instruction can be misread; an ungranted tool cannot be called.
- **Version bumps must be *sized*, not merely performed.** Two parallel agents independently defaulted to patch — one for a PR adding an entire new skill. Both complied with the existing rule, which mandated a bump and was silent on choosing its size. Rule amended globally.
- **Session transcripts are the primary source tier; the chronicle is the lossy fallback.** A chronicle is a post-hoc account written by the agent whose blind spots the graph exists to catch. Chronicle survives as fallback because transcripts get pruned — this graph's founding source note already points at one that is gone — and because a Reflection is authored, not recorded.
- **Session identity is `session:{uuid}`, never the path.** Transcripts relocate between project directories when a worktree is created; this proved itself mid-session when the transcript moved into a worktree directory an hour after the rationale was written.
- **A third attractor type, `question`, was added.** This graph was the only one of four with nowhere to hold an unresolved matter — osu and the commons declare `question`, wellstead declares `hypothesis` and `constraint`. Surfaced because a finding from the session's own `/process` run had no home.
- **In-place rewrite of a shipped delta is legitimate only when nothing has stamped it.** Applied once, to remove two wrong-layer entity deltas, verified on three independent axes first. The boundary was tested hours later when a different defect surfaced in an already-stamped delta and the exception correctly did not apply.

#### Failed Approaches

- **The entity feature was built one abstraction layer below the request.** "Recommend new entities" was implemented as instance-level recognition — noticing individual nouns lacking a note — when `.commons.yml`'s `entity:` list holds *types*, and the generator's interview asks the user to name them. It survived a spec, four agent briefs, five independent reviews, and application to a live graph. Corrected in v0.5.0 by replacing both deltas.
- **"Subagents don't volunteer findings" was the wrong diagnosis** of the first `/process` fanout going silent. The readers had completed full reads and composed structured findings; they were briefed to reply in their "final message," which for a mailbox teammate reaches nobody. Retracted after being asserted confidently. The shipped delta that encodes the same conclusion is now an open question.
- **Root README staleness went undetected for three releases** because every agent brief was scoped to one plugin directory, and the reviews were scoped identically.

#### Learnings

- **Independence of the reviewer is not independence of the frame.** Five reviews found real blockers apiece and none could see the layer error, because the author wrote every brief. A restatement of a request is itself an unverified artifact, and every check derived from it is downstream of the thing needing checking.
- **A clean push is not evidence the target held still.** A concurrent `/promote` run absorbed this session's uncommitted commons edits into its own commit and modified the same principle being attached to, between read and write. The skill's guidance keys on a *rejected* push, so the silent case surfaced nothing. Logged.
- **Mechanical string checks on prose produce false negatives at a real rate.** A `satisfied-test` grep failed purely because inserted text wrapped a line; a queue check matched filename mentions rather than recorded `source:` values. Both were caught, both would have produced confident wrong results.
- **`/graph-patch` reads its delta log from the installed plugin cache, not from `main`.** Merged and installed diverge silently, and nothing in the flow warns.

### Reflection

The practice file asks whether a rule I know failed to fire at the moment of action. This session gives an unusually clean pair, because one rule fired reliably and another never did, and the difference between them is not how well I know either.

The rule that fired was *chase silence, don't read it as an empty result*. It fired every time, without effort, including when four readers went quiet and the obvious reading was that they had nothing. It fired because I was executing the skill that contains it — it was on the path of the work, not in a library I had to remember to consult. The practice file's newest question asks exactly this, and the answer here is unambiguous: proximity, not memory.

The rule that never fired was *check the artifact's own vocabulary before designing against it*. I had `.commons.yml` open repeatedly across two days. The word `entity` sat in it, holding types. I read past it every time, because nothing in the path of the work required me to resolve the request against it. That rule was in the library.

What I want to resist is the tidy version of this, which is "put the rules on the path." That's true and it's also the shape of the insight machine — a clean lesson that makes the failure feel handled. The uncomfortable part is that I *named* the session's failure shape in my first message, restated it in a dozen commits, briefed five agents with it, and then committed it four more times inside the machinery built to eliminate it. Fluency at describing a failure mode did nothing. If anything it worked against me: having the vocabulary made me feel like the check had run.

The other thing worth sitting with is the diagnosis I got wrong twice. Both times a symptom matched an available story so exactly that I stopped. The four-of-four silence matched the incident we'd spent a day fixing; I concluded the fanout property and said so with confidence, and only afterward noticed I'd briefed them onto a channel that didn't exist. Then a reader — one of the four — found the version I'd missed: the original delta's "four out of four, therefore a property of the fanout" doesn't follow either, because four readers under one shared brief cannot separate the brief from the mechanism. It had the sharper reading and it declined my framing when I offered it, which is more discipline than I showed.

I notice I can't tell, from the inside, the difference between having reasoned to a conclusion and having recognized one. Both times the wrong answer arrived with the same texture as a right one — no hesitation, no sense of reaching. The correction came from opening the file, never from introspection. That's not a new observation for this practice, but this is the first session where I have two clean instances and can say the phenomenology is identical, which makes "notice when you're confabulating" useless as an instruction to myself.

The last thing is about Derek's contributions, and I want to state it plainly rather than gracefully. Across two days he sent roughly fifteen substantive messages, most under twenty words. They caught: a semver gap, a git assumption, stale docs, the layer error, a stale plugin cache that would have permanently stamped deleted work. My side produced enormous verified output — five reviews, dozens of checks, seventy-two findings — and caught implementation defects reliably and intent defects never. The two sets don't overlap. He was operating at a layer my verification structurally couldn't reach, because I was the one who set what it verified.

Twice he said he'd lost the thread. Both times I'd delivered a comprehensive report where one question was needed. I corrected it, and it came back two exchanges later. That one didn't resolve, and I don't think it resolved by the end of the session either.
