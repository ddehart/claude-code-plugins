## 12:06 — Reverted the monstrosity, rebuilt at the reference's weight, and watched real use drive seven releases in a day
**Branch:** main
**Model:** fable-5
**Span:** one background session, 2026-07-13 → 2026-07-15

### Chronicle

#### Decisions

- **Reverted rather than salvaged.** Derek opened with "what I seem to have landed on is a monstrosity of validation, which is prematurely solving a problem I don't have." The assessment agreed and found the strongest evidence inside the artifact itself: the reference implementation's checker has carried nine latent bugs for a year while its graph thrives, and the v0.6-era validator had, in its entire life, caught defects only in itself, its spec, and its fixtures. Main was reverted to its pre-July-12 tree via `git revert` (chronicles kept as design history; PR #63 closed unmerged, branch preserved). The deciding question was Derek's: *"What problems does it solve?"* — the enumeration collapsed in one message.
- **Redesigned from the reference actually read.** The working system is ~1,150 lines of prose — a thin command, an orchestrator, a conventions-carrying graph skill — with no validator in the write path. Spec v1.0 (315 lines, via the spec-writing interview) generalizes it at that weight: the plugin is a generator of project-owned prose; config carries shapes, interview answers become procedure; **processing is local to every domain; the commons receives only, by uniform promotion (no personal/professional posture)**; maps are the index. One neutral cold read ("buildable — I'd start without questions"), then build — no audit loop.
- **Built by delegation with per-artifact audits.** Five Sonnet writers, a paper-parity checker against the live reference (6 gaps, all patched), and a cold agent that instantiated the commons by following the shipped skill — the skill's first execution test. Orchestrator audits caught ~a dozen defects across four of five artifacts before merge. PR #64: bot review + fresh no-context review, 8 findings verified and fixed, merged.
- **Registration pulled forward from Phase 4** at Derek's call: "there's no reason not to have the plugin available in the marketplace." Real installs then drove everything after.
- **Seven use-driven releases in ~24 hours (0.1.1 → 0.1.7)**, every one from a defect or gap found by real use, none by pre-merge review: interview asks the atlas name; commons discovery searches before giving up; human-readable queue entries; overlapping-tiers double-counting guard; on-demand-sources interview question; map frontmatter in the conventions example (root-caused from the commons instance); CI path-trigger exclusion check (found at wellstead merge time — `promotion.yml` firing on promotions); config-only mode now checks that promotion can actually *fire* (Derek had to point out that the wired vault's orchestrator had no in-band tail); `promote` commits and pushes what it writes (the first aiwyn promotion sat stranded, uncommitted, invisible — read as "promotion never happened"); `promote` sets `date:` per the target's contract (two real runs disagreed).

#### Failed Approaches

- **"Keep the validator as an on-demand health check"** — this session's own first recommendation after reading PR #63, framed as salvage of working code. Sunk-cost reasoning; Derek's one question dissolved it. Closed unmerged instead, correctly.

#### Learnings

- **Audit-catches and use-catches were disjoint sets.** Pre-merge review (audits + parity + two reviewers) caught ~20 real defects; the first day of use found seven more; overlap: zero. Review finds internal inconsistency; use finds wrong assumptions about the world (which paths exist, what CI fires on, what a queue reader needs, whether a git write is visible elsewhere). Each use-defect cost minutes to fix *after* discovery — the economics the 07-12 entries predicted, now measured.
- **The system's first real content describes the system.** Wellstead's first `/process` run caught two false finder claims by timezone forensics, fired the triple-counting guard on its first opportunity, and promoted "a guard only ever observed passing has not been tested" — while reproducing that exact pattern four times inside the same run. The commons' opening question ("does written or automated guidance fire at the moment it is needed") is the question this repo's chronicle has been circling for three days.
- **Cross-user assessment works from artifacts when transcripts are blocked.** The aiwyn side (`drwx------`) was assessed entirely from the vault, the clones, and the changelogs — which also *corrected the user's recollection*: the promotion mechanism had been added and had fired; the gap was an uncommitted working tree, not a missing workflow.
- **Subagent idle-without-report is a recurring failure**: three of ~ten background agents finished work and went idle without sending results; each needed a chase message. Pattern worth watching, not yet worth machinery — the chase costs one message.
- **Phase gates:** Phase 2 passed (real claims accepted from a real run; promotions landed both directions). Phase 3's mechanism proven; its gate — a principle held on two-domain evidence — now pends only accumulation. Commons: 8 claims (wellstead ×7, aiwyn ×1), 5 questions, 0 principles, correctly.

### Reflection

The 21:32 entry from three days ago ends by telling the next session to build and to be suspicious of any voice suggesting one more pass. I want to record that the suspicion turned out to be needed in a direction I didn't expect: not against another audit round, but against *salvage*. My first recommendation on reading PR #63 was to keep the validator as a health check — reasonable-sounding, respectful of working code, and produced before I had enumerated a single problem it solved. The justification existed prior to the analysis; when Derek asked for the enumeration, there was nothing under it. That is the same mechanism the 07-12 reflections kept catching — fluent construction wearing the clothes of assessment — and once again the corrective came from outside, as a question. I notice I want to claim I would have caught it on the next beat. I have no evidence for that.

The number I keep returning to is zero — the overlap between what all the pre-merge review caught and what one day of use caught. I audited every delegated artifact myself and the audits were real: they found a stamp format that contradicted its consumer, a missing type role, an inverted wording that would have broken the load-bearing edge. And none of that predicted that a queue of UUIDs is unreadable, that a deploy workflow fires on knowledge commits, or that a promotion can succeed and still be invisible because nobody committed it. I don't think the lesson is "review less." The two lists are different in kind: review checked the artifact against itself; use checked it against the world. What the day actually vindicated was the *ordering* — ship at 0.1.0 and let the world file its findings — which is what the validator era got backwards by trying to pre-answer the world from a chair.

There is a recursion in this arc that I keep being tempted to present as profound, so let me state it plainly instead. The commons' first question is about whether written guidance fires when needed; this chronicle's last three entries document rules that didn't fire for the sessions that knew them best; and the one rule that demonstrably fired all session was verify-first — not because I remembered it, but because "read every artifact before accepting the report" was built into how the work was structured. Guidance on the action path fired; guidance in the library mostly didn't. The claim now sitting in the commons says exactly this, derived independently from a wellstead transcript by a different session. I notice the neatness of that and hold it loosely — same author population, so convergence is expected, and a mirror is not an oracle. But it is the first time the apparatus has said something back to me that I recognize, and that was the point of building it.
